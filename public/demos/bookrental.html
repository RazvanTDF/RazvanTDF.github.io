BookRental — a library rental system built with Spring Boot.

A REST backend in Java 21 + Spring Boot 3 with PostgreSQL, Hibernate ORM, scheduled background jobs and email integration. Built end-to-end as part of the Java Academy training program.

Java 21 + Spring Boot 3 PostgreSQL + Hibernate REST API tested with Postman Docker for local services
01 · Project

What the application does

BookRental manages a complete book rental system for libraries: users, book copies, rental workflows with state transitions, automatic email notifications, and scheduled jobs that run in the background.

Catalog

Books, authors, publishers and libraries — each book with multiple copies distributed across different libraries.

Rentals

Full rental workflow with status transitions: PENDING → IN_PROGRESS → FINISHED / DELAYED / CANCELED.

Notifications

Automatic emails for confirmations, reminders and cancellations. Asynchronous dispatch with @Async.

Scheduled jobs

Cron jobs that cancel unclaimed reservations, flag overdue rentals, and reassign copies in case of conflicts.

Search & Pagination

Endpoints with optional filter parameters and configurable pagination for all major resources.

Strict validation

DTO-level validation (Jakarta Validation) plus business rules in services with correct HTTP status codes.

02 · Stack

Technologies used

The stack was chosen to reflect a realistic enterprise Java backend setup.

Java 21 Spring Boot 3 Spring Data JPA Hibernate ORM PostgreSQL Jakarta Validation Spring Mail (SMTP) Spring Scheduler Spring Async Gradle Docker Postman Git JWT / Keycloak (team-provided)

Persistence

PostgreSQL running in Docker, accessed through Spring Data JPA and Hibernate. Full relational mapping: @ManyToOne, @OneToMany, @ManyToMany with join tables, cascade and orphan removal.

Tooling & Testing

Postman for REST endpoint testing, with collections organized per resource. Docker used for running PostgreSQL and Keycloak locally during development.

03 · Architecture

Layered architecture

Clear separation of concerns across layers, each with a single responsibility — the standard pattern for Spring backends.

Controller @RestController · accepts HTTP requests, validates input, delegates to the service
↓
Service @Service · business logic, @Transactional, orchestrates repository + mapper + email
↓
Mapper @Component · converts Entity ↔ DTO, no business logic
↓
Repository JpaRepository · derived queries, custom JPQL with optional filters, pagination
↓
Entity @Entity · mapped to PostgreSQL tables, field-level validation, JPA relationships

Isolated schedulers

Separate @Scheduled classes that only trigger service methods — the business logic stays in the service layer.

DTO at the boundary

Entities never leave the service layer. Everything exposed through the API is a validated, explicitly shaped DTO.

04 · Features implemented

What I built personally

Functionality I wrote end-to-end: from entity and repository all the way to REST endpoint tested in Postman.

01

Complete CRUD across all entities

Book, Author, Publisher, Library, Librarian, User, Exemplar, Rental — each with its own DTO, Mapper, Service and Controller.

02

Rental status workflow

A Status enum with an isNextStatePossible() method that validates allowed transitions between PENDING / IN_PROGRESS / FINISHED / DELAYED / CANCELED.

03

Cron-based schedulers

Pickup reminders for pending reservations, automatic cancellation after 3 days, overdue flagging, and copy reassignment on conflict.

04

Asynchronous email dispatch

SMTP integration via Spring Mail with @Async for reservation confirmations and status-change notifications.

05

Pagination & search

/list endpoints with PageRequest and configurable sorting, plus /search endpoints with optional filters combined in JPQL.

06

Two-level validation

Jakarta Validation on DTOs (@NotBlank, @Email, @Min) plus business rules in services using ResponseStatusException for correct 404 / 400 responses.

07

Full JPA relationships

ManyToOne (Book ↔ Publisher), OneToMany (Book ↔ Exemplar), ManyToMany with join table (Book ↔ Author), with cascade and orphan removal configured.

Transparency note: the Spring Security + Keycloak (JWT) integration was provided by the training team as shared infrastructure. I worked with it (tokens, roles, resource server configuration) but I don't claim it as my own implementation.
